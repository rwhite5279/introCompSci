1. Beginnings, and HTML

Let's get started, shall we?

1.a. A Brief History of the Internet

The World Wide Web (WWW) didn't spring into existence overnight. The first email sent from one user to another (from within a single machine) occurred in 1965. It wasn't until a few years later, in 1969, that the ARPAnet, precursor to the Internet, sent and received its first packets of information.

You can read more about these early days of the Internet's development on Wikipedia, and hear a bit about it from Ethan Zuckerman's A History of the Internet (2 minutes, YouTube).

Note that the Internet—a network for passing information back and forth, is NOT the same thing as the World Wide Web. That was yet to come.

1.b. The World Wide Web

In 1989, Tim Berners-Lee famously proposed a system by which hypertext viewed in a browser could be used "to link and access information of various kinds as a web of nodes in which the user can browse at will."

To accomplish such a thing, two devices were needed: a webserver to deliver the information (first built in 1990), and browser software to allow one's computer to traverse the hyperlinks.

The first graphical browser, Mosaic, was created in 1993.

You can read more about the development of the World Wide Web on Wikipedia.

HyperText Markup Language provides a means of coding webpages so that they can be correctly displayed by a browser. It uses tags (indicated by angle brackets) to indicate how text‚Äîstructured in the form of headers, paragraphs, lists, embedded graphics, etc‚Äîshould be rendered.

HTML was initially proposed by Berners-Lee in 1990, and published in 1993, modified to HTML 2.0 in 1995...

<!-- This is an example of a webpage --> <html> <head> <title>Mr. White's Web Page</title> </head> <body> <h1>A Web page</h1> <p>If you want to see my website, please <a href="http://www.crashwhite.com">click here</a>.</p> </body> </html>

Webpages are most often viewed using a browser: Microsoft's Internet Explorer, Mozilla's Firefox, Safari, Chrome, Opera, etc. The HTML source code for the page is interpreted by the browser and then displayed.

You can see the source code for most webpages by clicking View > Source in your browser's menu.

Creating a webpage is easily done by creating a text file with the extension .html; opening that file using a browser will cause the browser to interpret the HTML code and display the page appropriately.

1.d. Writing a webpage

Let's get started.

1.d.i. Create your workspace

Create a new folder in your compsci directory call public_html . All of your webpages will be stored in this folder.

$ cd ~ $ mkdir public_html

Now, navigate to that folder and create the following file in it.

$ cd public_html $ nano helloworld.html <!-- My first webpage --> <html> <head> <title>"Yup, it's me!"</title> </head> <body> <p>Hello, world!</p> </body> </html>

Once you've got it saved, use a browser (Firefox or Safari) to File > Open file... the helloworld.html document.

1.d.ii. Create a link

The real power of "the web" isn't displaying text... it's using hyperlinks to navigate from one page to another.

Edit your file as follows, and then test it in a browser to see how it works.

<body> <p>Hello, world! <!-- Line break? --> Would you like to read <a href="http://www.latimes.com">the news?</a> </p> <p>You don't <strong>have</strong> to!</p> </body>

Note that the spacing of the text in the html code doesn't necessarily match the spacing of the text when it's displayed in the browser. While tags give you some control over formatting, in general html is designed so that text can be flexibly displayed in a browser window, whether it's on a Mac, a PC, a tablet, or a phone.

1.d.iii. Tags

A list of some of the most common HTML tags:

  • <html> ... </html> encloses HTML code
  • <head> ... </head> encloses header information
  • <body> ... </body> encloses the code you'll be using to display information
  • <!-- encloses a comment -->
  • <p> ... </p> enclose paragraphs
  • <br /> inserts a break
  • <b> ... </b> create bold text
  • <i> ... </i> create italicized text
  • <a href="thesecondpage.html">Page 2</a> encloses a link
  • <img src="my_photo.jpg" /> inserts an image

2. Cascading Style Sheets, sftp

Take a look at the following code, and guess what gets displayed on the screen when this page is displayed in a browser?

<!-- This is an example of a webpage --> <html> <head><title>Just an example</title></head> <body style="background: #ccc"> <div style="font-size: 200%">Welcome!</div> <div style="font-style: italic">What would you like to do? <ul> <li>View my <a href="code.html" style="font-weight: bold">Python programs</a></li> <li>View my <a href="bio.html" style="font-weight: bold">biography</a></li> </ul> </div> <div style="background: yellow; color: #fc0">Updated 2010-03-23</div> </body> </html>

You can see that some styling has been added to this page.

Styles allow one to extend the formatting conventions of standard HTML, but they bring additional challenges with them. What is good style? How does a webpage designer ensure consistent style throughout a website? How can large-scale site changes be managed?

The solution is Cascading Style Sheets.

2.a. Cascading Style Sheets (CSS)

Cascading Style Sheets are a means of describing the formatting of various elements in an HTML file.

Let's see what that looks like for an actual webpage.

Although it's certainly possible to individually assign styles for each element of a webpage, that can a) get tedious, and b) lead to errors when changes to a style have to be made throughout a site.

Let's take a look at the .html file for this page.

You can see here some of the html tags that we've started using, but there's also an interesting reference to an external .css file. It's that file that contains all of the styling for our website.

What does that file look like?

You can see here that the .css file has references to all sorts of formatting rules that are different for different elements or sections of the document. Margins, padding, background colors, text colors, widths, and other specifications have all been defined for the different parts of the html document.

One of the primary goals of CSS is to separate a website's content from its formatting. The .html document describes the content of the site, while the .css document describes the formatting of the site.

Create two new webpages and a third file, a custom style sheet, that is used by both pages to produce consistent styling across both pages.

Your webpages should refer to the css in the header as follows:

<style type="text/css"> <!-- @import url("screen.css"); --> </style>

You can specify formatting rules for already defined elements of a webpage: paragraphs (<p>), headers (<h1>), etc.

You'll also have divisions in the body of your webpage that you can indicate as follows:

<div class="shout">Everything in here will be part of a "shout"</div>

Your CSS file should be called screen.css, and include style descriptions similar to these:

html {margin: 0; padding: 0; } body {margin: 0; padding: 0; border: 0; font-family: Verdana, Arial, Helvetica, sans-serif; font-size: 80.01%; font-weight: normal; color: #000; background: #ccc; } // Defaults for the whole page p {font-style: italic} // Sets all paragraphs to italic div.shout {font-weight: bold} // All text in a division called "shout" // will be bold a:link, a:visited {text-decoration: underline; color: red; } a:hover, a:active {text-decoration: underline; color: blue; background-color: #cf0; } // These lines describe how "anchors" (links) behave

2.b. The HTTP server

Up to this point, your webpages have only been visible to (and usable by) you. For them to be visible by others, they need to be uploaded to a webserver, a program that runs on a networked computer and is specifically designed to deliver pages on the WWW.

The crashwhite.polytechnic.org server that we've been using to store our files, and run Python, is also running Apache, a popular open-source server.

When someone on the Internet types in a Uniform Resource Locator (URL), it usually includes a Domain Name such as polytechnic.org or crashwhite.com. A Domain Name Server (DNS) keeps a list of what domain names are mapped to what Internet Protocol (IP) addresses, and routes requests for your webpage to the appropriate server.

For the moment, there aren't any terribly useful html files on our server, so you won't actually be able to see anything, but... let's point our browser to the server and see what happens anyway.

Launch Firefox and enter the Uniform Resource Locator (URL): http://crashwhite.polytechnic.org/~rwhite

This goes to the folder public_html in /home/rwhite/ and tries to find a file there that it can display (default index.html).

What we need, then is a way of getting your local .html files that you've written onto a webserver, so that people looking for your website will be able to see your webpages.

2.c. Secure File Transfer Protocol

It may be that you creating your .html files on the same server that's going to be serving them. If so, you don't need to worry about transferring them to the server--they're already there.

More commonly, though, you're working offline or on a different computer, and you need to be able to transfer the appropriate files to the server at a later time. We've already learned how to use SecureCopy (scp) to do this. Now we're going to learn another—and for our purposes here, more useful way—to transfer files.

Assuming you have a webpage written that you'd like to serve, you need to upload it to the server using Secure File Transfer Protocol (SFTP, included with SSH).

The basic syntax for logging in to the server using sftp looks like this:

$ sftp rwhite@crashwhite.polytechnic.org Connecting to crashwhite.polytechnic.org... rwhite@crashwhite.polytechnic.org's password: sftp> ls Documents Public public_html sftp> cd public_html

Once you've logged in using sftp you can issue commands, much as you would if you were logged in via ssh. Although there are very similar commands in some cases, ssh and sftp are used for different things.

A comparison of some ssh and sftp:

SSHSFTP
Used for working on the serverUsed for transferring files to and from the server
$ ssh rwhite@crashwhite.polytechnic.org$ sftp rwhite@crashwhite.polytechnic.org
$ pwd # prints working directory$ pwd # prints remote working dir
$ lpwd # prints local working dir
$ ls # lists the files in a directory$ ls # lists files in remote directory
$ lls # lists files in local directory
$ cd ../ # changes directory$ cd ../ # changes remote directory
$ lcd ../ # changes local directory
$ get filename.html # downloads file from current remote directory to current local directory
$ put filename.html # uploads file from current local directory to current remote directory

So, let's say you have a couple of pictures on your local machine in the directory Pictures, and you want to upload them to the server so that you can use them in a webpage. What would that look like?

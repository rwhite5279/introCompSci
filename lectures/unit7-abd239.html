Introduction to Version Control with Git

Once you begin writing more complicated programs, you'll find it helpful to have some sort of Version Control System.

A VCS helps you keep track of ongoing progress as you develop your program, and allows for multiple versions of your program to exist simultaneously. What if you make edits to a program over the course of a day or three, only to find that a previous version of the program works better than the new one? What if you're sharing development of a program with a partner or a team, with the understanding that different people will be writing different parts of the program. Who keeps track of the "official" version of the program? How do the members of a team combine work?

You do not want to be the loser in charge of copy-pasting snippets of code and proofreading/debugging thousands of lines of code.

Version Control helps to manage complex projects. There are different systems one can choose from, including SVN, Subversion, CVS, and Git. Because of the recent popularity of GitHub, we're going to be learning Git.

1. Before Using Git

Let's set up a directory that's going to store the files that we're working on. It could be a public_html folder that holds all the files for a website, or it could be be a myProject directory that holds a graphics-based game you're working on, as well as the graphics.py module that needs to be imported into that game. Regardless, you've got a folder, and for the purposes of our discussion, we're going to call it myProject.

2. Install Git

Git is a software package that runs via the command line (Terminal). To install it:

  • Mac OS X - You almost certainly already have git installed. Try typing git --version to see.
    $ git --version git version 1.7.4.4
  • Linux - You'll probably need to install git-core, which is easy enough to do. On Ubuntu for example:
    $ git --version The program 'git' is currently not installed. You can install it by typing: sudo apt-get install git-core $ sudo apt-get install git-core . . .
  • Windows - Download and install the msysgit package.

Once you've got Git installed on your machine, give it some global characteristics that it can use to identify you when you're working on your projects:

$ git config --global user.name "Richard White" $ git config --global user.email "rwhite@crashwhite.com"

2. Using Git

There are a few different ways to use Git. We're going to start out by seeing how to use it to manage ongoing progress of your own local work—you're basically working alone, and just using git to keep track of multiple versions of your project as you develop it.

Then, we'll see how you can "check out" a software version from a central repository, develop that package, and then commit your changes back to the repository.

2.a. Initializing Git with a local directory

It's super easy.

  1. At the command line, navigate into the directory of project.
  2. Type git init

That's it. Let's see what happens before and after.

$ cd myProject $ ls -a . README.txt rogue_notes.txt .. rogue.py $ git init Initialized empty Git repository in /Users/rwhite/Desktop/myProject/.git/ $ ls -a . .git rogue.py .. README.txt rogue_notes.txt

Awesome. You can see that a new "hidden" directory has been created. (It's ordinarily hidden from the ls command, unless we specify the -a flag to show all contents). It's this .git directory that holds all the information that will help us manage the multiple versions of our program over time.

Note that we've initialized git, but we haven't actually done anything with it yet.

2.b. Adding files, Committing files

At this point, my files are untracked—they're in the directory that I'm working on, but I haven't yet told Git to start monitoring them. You can check the status of all the files in your project using the git status command:

$ git status # On branch master # # Initial commit # # Untracked files: # (use "git add <file>..." to include in what will be committed) # # README.txt # rogue.py # rogue_notes.txt nothing added to commit but untracked files present (use "git add" to track)

I'm going to add all these files to the git index so that they can be tracked:

$ git add -A

Now what does the git status look like?

$ git status # On branch master # # Initial commit # # Changes to be committed: # (use "git rm --cached <file>..." to unstage) # # new file: README.txt # new file: rogue.py # new file: rogue_notes.txt #

Interestingly, Git still doesn't know anything about our project, at least not officially. You can check on your recent activity with git by checking out the git log:

$ git log fatal: bad default revision 'HEAD'

The last step, then, in making sure that Git knows about what we're doing is to officially commit our work to the Git index. "Committing" is along the lines of a large-scale "save," in the same way that one might save one's status in video or computer game: you don't want to do it every minute or two, because that's time consuming, annoying, and quickly fills up your disk with unnecessary data, but you definitely want to do it right before you take on a boss.

Let's do our first commit on this project:

$ git commit 1 2 # Please enter the commit message for your changes. Lines starting 3 # with '#' will be ignored, and an empty message aborts the commit. 4 # On branch master 5 # 6 # Initial commit 7 # 8 # Changes to be committed: 9 # (use "git rm --cached ..." to unstage) 10 # 11 # new file: README.txt 12 # new file: rogue.py 13 # new file: rogue_notes.txt 14 #

Each time you commit, you'll be asked to enter information regarding that commit, which will be used (by yourself or others) later on to identify what happened during this particular commit. It's recommended that you use a short (<50 characters) description of the changes that you've made, followed by a blank line and then a more complete version of your description, if desired.

At this point, you should try changing a file in the directory, then using git add and git commit to record changes to your project. Note that you can use the command git commit -a to combine these two steps.

Then, take a look at git log and see what it says:

$ git log

You'll see a listing of your changes to the project from most recent down to oldest, along with a list of who made those changes, and when. Not bad!

2.c. Comparing Versions

So far this may seem like a lot of work, and we haven't seen any real benefit from this version control. But here's where the benefit comes in.

2.c.i. Recovering from changes that you haven't committed yet

$ git reset --hard HEAD

2.c.ii. Returning to an earlier committed state

$ git reset --hard HEAD

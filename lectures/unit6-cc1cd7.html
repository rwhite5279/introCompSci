1. Some Final Comments on Programming and Design

We've covered a fair amount of ground in here in the last eleven weeks:

  • Working in BASH on the Terminal, locally and on the server
  • Intro to programming in Python, including output, input, control structures, and data structures
  • Organizing smaller Python units (functions, objects) into larger programs
  • Writing HTML code to produce static webpages
  • Intro to database design, and using Python to manipulate an SQLite database

We haven't wasted a whole lot of time talking about programming philosophy, or software design principles, or Big Picture stuff like that. (If you're interested in Real Life stories of coding, check out slashdot.com, or Coders at Work: Reflections on the Craft of Programming, by Peter Seibel.)

Perhaps there are a few principles, though, that you may have picked up on as we've worked on developing our skills:

  1. Programming is Hard
  2. Google is your friend
  3. Top-Down Design can help
  4. Bottom-Up Design can help
  5. Functions / Objects can help you structure your work
  6. Print statements are your friend

2. Programming Exercises

Before beginning a long-term final project for this class, let's spend a little time brushing up on a series of smaller programs that will allow us to review some fundamental techniques.

Programming Exercises

  • happy_birthday.py
    Write a program that prints the lyrics to the Happy Birthday song for every person in our class. The program should consist of two functions: one called "sing()" that prints the lyrics to the song based on a parameter "name" that is passed in, and a main function that uses the list "people" to store the names of everyone in the class, and then calls the sing() function with those names.
  • sieve_of_eratosthenes.py
    The "Sieve of Eratosthenes" finds all prime numbers up to n using this strategy:
    1. Create a list of all numbers from 2 to n.
    2. Remove first number the list, identify it as a prime number.
    3. Remove all multiples of this number from the list as well.
    4. Continue until the list is empty.

    Write a program that asks the user to enter a number n and then uses the sieve algorithm to find all primes less than or equal to n.
  • the_locker_problem.py
    There are 1000 students at a school, and each student has a locker, and the lockers are all in a line, and numbered from 1 to 1000. The lockers are all initially closed at the start of the problem, when student #1 starts at locker # 1, and visits locker, opening them. Student #2 starts at the second locker, and visits every other locker (2, 4, 6, etc.), and closes them. Student #3 starts at locker #3 and visits every third locker (3, 6, 9, etc), and closes the door if it's open, or opens the door if it's closed. This process continues until all 1000 students have traveled along the line of lockers.

    What is the final state of each locker in the row of 1000 lockers? Write a program to simulate the opening and closing of the lockers in this problem.
  • one_D_life.py
    From http://www.stephenwolfram.com/interviews/85-sciam/
    A one-dimensional cellular automaton (hereafter called a line automoton) consists of an infinite strip of cells changing states according to a given set of rules. As in the game of Life a cosmic clock ticks away and at each tick every cell enters a state determined by its previous state and the previous states of cells in its neighborhood. A line automaton is specified by giving two numbers called k and r as well as a set of rules for deriving the next state of a cell. The first number, k, determines how many states are allowed for each cell. In Life there are just two states and so k is equal to 2; among the line automata to be considered, higher values of k are common. The second number, r, refers to the radius of neighborhoods use to compute the next state of a cell. The present state of a cell and the states of its r neighbors on both sides determine the next state of the cell. For example, if r is equal to 2 and k is equal to 3, a certain rule might specify that when a cell's neighborhood looks like [0,2,1,1,0] the next state occupied by the central cell would be [3].

    The set of rules that define a line automaton must decided the fate of a cell for every possible configuration of states inhabiting its neighborhood. Depending on the size of k and r the number of possible rule sets to consider can be enormous... The number of possible rule sets to consider is greatly reduced by adopting the ones Wolfram calls totalistic. Here the next state of a given cell is determined only by the sum of the states in the cell's neighborhood. The sum includes the state of the given cell. The number of possible sums varies from 0 to m, where m is the largest value of the state multiplied by the size of the neighborhood. If one specifies how these sums become the central cell's next state, one has specified the line automaton completely.

    For example, there is a very interesting line automaton governed by totalistic rules given in this table:

               +-----------------------+
       sum | 5 | 4 | 3 | 2 | 1 | 0 |
           +-----------------------+
next state | 0 | 1 | 0 | 1 | 0 | 0 |
           +-----------------------+
    Write a program to track a line automaton, by:
    1. Creating a list of an initial state of one hundred zeros and ones and print it out.
    2. Apply the rules described in the rule table given above to find the next state and print it out.
    3. Repeat.

    This program uses an external file to keep track of a grocery store shopping list. The program should allow the user to A)dd to the list, S)how the list, D)elete a single item from the list, and C)lear the list.
